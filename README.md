# Geo-Grouping Telegram Bot

Телеграм-бот, который собирает геолокации пользователей и объединяет их в группы
по 3 человека, находящихся близко друг к другу (в пределах заданного радиуса).
Когда группа собрана — все участники получают уведомление. Состояние видно на
живой веб-карте, где можно вручную пересобрать группы под самых ближних соседей.

## Что делает проект

Перед отправкой геолокации каждый пользователь проходит **онбординг**-мастер:

1. `/start` → выбор языка: **английский**, **русский** или **узбекский** (латиница).
2. Выбор **кампуса**: Мирзо Улугбек (41.356250, 69.373209) или Яшнобод (41.256928, 69.328708).
3. Передача **номера телефона** через кнопку «Поделиться контактом» (Telegram requestContact).
4. Отправка **геолокации** через кнопку «Отправить геолокацию».

После онбординга:

5. Бот добавляет точку в очередь ожидающих своего кампуса и сразу пытается собрать группу.
6. Как только находятся **3 человека одного кампуса, все попарно в пределах радиуса** (по
   умолчанию 5 км) — формируется группа, и все трое получают уведомление со
   списком соседей и расстояниями. **Группировка происходит только внутри одного кампуса** —
   участники разных кампусов никогда не попадают в одну группу.
7. Если участник выходит (`/leave`), его группа распадается: остальным приходит
   уведомление, они возвращаются в очередь, и поиск запускается заново.
8. На веб-карте (`/live`) видно всё состояние в реальном времени; кнопкой
   «Пересобрать группы» можно глобально переоптимизировать группы так, чтобы у
   каждого были самые ближние соседи — **не оставив никого в одиночестве**.

Для тестов включён `TEST_MODE`: один Telegram-аккаунт может отправить несколько
локаций, каждая считается отдельным участником (можно собрать группу в одиночку).

## Как запустить

```bash
npm install
cp .env.example .env        # впишите BOT_TOKEN от @BotFather
npm run bot                 # запускает Telegram-бота И живую карту на :8080
```

- Telegram: напишите боту `/start`.
- Карта: откройте http://127.0.0.1:8080/live

> `npm run bot` поднимает один процесс, который обслуживает и Telegram (long
> polling), и живую карту с эндпойнтом пересборки. Это сделано намеренно: бот —
> единственный владелец состояния, поэтому веб-кнопка пересборки работает с тем
> же состоянием, что и бот, без гонок за файл.

## Архитектура

Один процесс (`npm run bot`), модули с чёткими границами:

| Файл | Ответственность |
|------|-----------------|
| `src/bot.ts` | Точка входа. grammY-хендлеры онбординга, команд и локаций; поднимает HTTP-сервер карты. |
| `src/config.ts` | Чтение и валидация `.env` → типизированный конфиг. |
| `src/store.ts` | Владелец состояния (`BotState`) в памяти; атомарная запись в `data/state.json`; оркестрация `joinAndMatch` / `leave` / `rebuild`; профили (`getProfile`/`setProfile`). |
| `src/matcher.ts` | **Чистые** функции группировки: `findGroups` (инкрементальный поиск) и `optimizeGroups` (пересборка). Геометрия — `haversineKm`, `computeCentroid`. |
| `src/grouping.ts` | Гео-утилиты: расстояние по формуле гаверсинуса и центроид. |
| `src/notify.ts` | Отправка сообщений в Telegram (с проглатыванием ошибок) и форматирование. |
| `src/i18n.ts` | Строки интерфейса на трёх языках (`en`/`ru`/`uz`); функция `t()` с интерполяцией. |
| `src/campuses.ts` | Константы двух кампусов с координатами; функция `campusById`. |
| `src/onboarding.ts` | Чистый автомат онбординга: шаги `language → campus → phone`, функции `startOnboarding`/`advance`. |
| `src/types.ts` | Общие типы: `BotState`, `Participant`, `Group`, `UserProfile`, `Language`. |
| `src/viewer-server.ts` | Фабрика HTTP-сервера карты (статика + `POST /rebuild`). |
| `viewer/live.html`, `viewer/live.js` | Живая карта (Leaflet), опрос состояния каждые 5 с, кнопка пересборки, маркеры кампусов. |

**Поток данных при новой локации:**

```
Telegram → bot.ts (message:location)
        → store.joinAndMatch(point)         // добавить + найти группы
            → matcher.findGroups(waiting)    // чистый поиск ближайших троек
        → store.save()                       // атомарно записать state.json
        → notify (если группа собралась)     // сообщения всем троим
```

Карта читает `data/state.json` по HTTP и перерисовывается; кнопка пересборки шлёт
`POST /rebuild` → `store.rebuild()` → `save()`.

```json
      "groupId": "group_001",
      "memberIds": ["u_007", "u_008", "u_009"],
      "centroid": { "lat": 41.31, "lng": 69.28 },
      "createdAt": "2026-06-17T10:01:00.000Z"
    }
  ],
  "profiles": {
    "123": { "language": "ru", "campusId": "yashnobod", "phone": "+998901234567" }
  },
  "campuses": [
    { "id": "mirzo_ulugbek", "nameKey": "campus.mirzoUlugbek", "lat": 41.35625, "lng": 69.373209 },
    { "id": "yashnobod",     "nameKey": "campus.yashnobod",    "lat": 41.256928, "lng": 69.328708 }
  ]

```

- `seq` — монотонный счётчик для генерации id (`u_001`, `group_001`, …). Счётчик

  Хранятся между сессиями: повторный `/start` перезапускает онбординг и перезаписывает профиль.
- `campuses` — список кампусов с координатами; перезаписывается из кода при каждой загрузке,
  поэтому карта показывает маркеры кампусов даже до первого участника.
- Запись атомарна (временный файл + `rename`) и сериализована через очередь
  промисов, чтобы параллельные хендлеры не повредили файл.

## Алгоритм поиска ближайших точек

Это ядро проекта — функция `findGroups` в `src/matcher.ts`. Она **чистая** (не
меняет вход, без побочных эффектов), поэтому легко тестируется.

### Задача

Из списка ожидающих участников собрать максимум групп ровно по `GROUP_SIZE`
человек (по умолчанию 3), где **все попарные расстояния внутри группы ≤
`GROUP_RADIUS_KM`** (по умолчанию 5 км). «Все попарно» — важно: близость к
центру/сиду недостаточна, каждый должен быть близок к каждому.

### Расстояние

Используется формула **гаверсинуса** (`haversineKm`) — расстояние по большому
кругу между двумя координатами на сфере Земли (радиус 6371 км). Это даёт
честное расстояние «по прямой» в километрах, корректное и около полюсов, и для
больших разнесений по долготе.

### Жадный алгоритм (nearest-neighbour)

```
findGroups(waiting, radiusKm, groupSize):
    pool = waiting, отсортированный по createdAt (кто раньше — тот раньше сид)
    used = {}                       # уже распределённые
    result = []

    для каждого seed из pool (по порядку):
        если seed уже used — пропустить
        candidates = все из pool, кроме seed и кроме used
        members = buildGroup(seed, candidates, radiusKm, groupSize)
        если members найдены:
            пометить всех members как used
            result.push({ memberIds, centroid: computeCentroid(members) })

    вернуть result
```

`buildGroup` — как нарастить группу вокруг сида:

```
buildGroup(seed, candidates, radiusKm, groupSize):
    # 1) оставить только тех, кто в радиусе от сида, и расстояние посчитать ОДИН раз
    near = candidates
           .map(c → { c, d: haversineKm(seed, c) })
           .filter(d ≤ radiusKm)
           .sort(по d возрастанию)          # ближайшие первыми
           .map(→ c)

    # 2) жадно добавлять ближайших, но только если новый кандидат
    #    в радиусе от ВСЕХ уже взятых членов (а не только от сида)
    members = [seed]
    для c из near:
        если len(members) ≥ groupSize: стоп
        если для всех m из members: haversineKm(m, c) ≤ radiusKm:
            members.push(c)

    # 3) группа годится, только если набралось ровно groupSize
    вернуть members, если len(members) == groupSize, иначе null
```

### Почему именно так

- **Сид по `createdAt`** — справедливость FIFO: кто дольше ждёт, того
  обслуживаем первым. Это делает результат детерминированным.
- **Сортировка кандидатов по расстоянию до сида** — берём сначала самых близких,
  то есть стремимся к плотным группам.
- **Проверка «в радиусе от всех членов»** (`okWithAll`) — гарантирует, что
  итоговая группа компактна целиком: любые двое в группе не дальше радиуса.
  Без этой проверки можно было бы собрать «цепочку», где крайние далеко друг от
  друга.
- **`used`-множество** — каждый участник попадает максимум в одну группу за
  прогон.
- Если у сида не набирается полная группа — он **остаётся ждать** (не попадает в
  result), и алгоритм переходит к следующему сиду.

### Пример

Радиус 5 км, группы по 3. Ожидают (lat, lng), ~0.01° ≈ 1.11 км:

```
a (41.300, 69.280)   b (41.301, 69.280)   c (41.300, 69.290)   ← все ~1–1.4 км
f (41.500, 69.280)                                              ← ~22 км от a,b,c
```

- Сид `a` (раньше всех). Кандидаты в радиусе: `b` (~0.11 км), `c` (~0.84 км),
  `f` отфильтрован (22 км > 5). Берём `b`, затем `c` — проверяем `b–c` (~1.4 км ≤
  5) → группа `[a, b, c]`.
- Сид `f`: ближе 5 км никого нет → группа не собирается, `f` остаётся ждать.

Итог: одна группа `[a, b, c]`, `f` в очереди. Как только подъедут ещё двое рядом
с `f` — соберётся и его группа.

### Когда запускается

`findGroups` вызывается **инкрементально**: после каждой новой локации
(`joinAndMatch`) и после каждого распада группы (`leave`). То есть группы
собираются «на лету» по мере поступления людей.

### Сложность

Для `n` ожидающих — порядка `O(n²)` вычислений расстояний на прогон (для каждого
сида перебираем кандидатов). Для десятков-сотен участников это мгновенно;
оптимизировать не требуется.

## Алгоритм пересборки групп

Кнопка «Пересобрать группы» на карте запускает `store.rebuild` →
`optimizeGroups` (тоже чистая функция в `src/matcher.ts`). Это **глобальная**
переоптимизация: в отличие от инкрементального `findGroups`, она пересматривает и
уже собранные группы, чтобы у каждого оказались самые ближние соседи — но с
жёсткой гарантией.

**Гарантия:** ни один уже сгруппированный участник не остаётся в одиночестве.

Итеративный жадный ребилд с «запиранием» исходных групп:

```
optimizeGroups(participants, existingGroups, radiusKm, groupSize):
    previouslyGrouped = { id участников со status == 'grouped' }
    lockedGroups = []        # исходные группы, которые решили не трогать
    lockedIds = {}

    повторять:
        pool = все participants, кроме lockedIds
        tentative = findGroups(pool, radiusKm, groupSize)   # тот же жадный поиск
        placed = все id из tentative

        # кто из ранее сгруппированных «выпал бы» (остался один)?
        orphaned = previouslyGrouped, кого нет ни в locked, ни в placed

        если orphaned пуст:
            # фиксируем: запертые исходные группы + новые жадные
            вернуть { groups: lockedGroups + tentative,
                      waitingIds: только бывшие ожидающие }

        # иначе запираем ИСХОДНЫЕ группы всех осиротевших целиком
        # (их участники выходят из pool) и пробуем снова
        для каждого id из orphaned:
            запереть его исходную группу
```

Свойства:

- **Лучший случай** — никто не осиротел: все группы пересобрались плотнее
  (каждый с ближайшими).
- **Худший случай** — заперлись все исходные группы: ничего не изменилось (что
  безопасно).
- На каждой итерации с осиротевшими запирается хотя бы одна новая группа →
  алгоритм завершается. Если у «сгруппированного» участника вдруг нет исходной
  группы (рассинхрон состояния), функция бросает ошибку, а не зацикливается; при
  этом `store.rebuild` сначала всё вычисляет и лишь потом применяет, так что
  состояние при ошибке не портится.
- Уведомления при пересборке **не отправляются** — изменения видны только на
  карте.

## Команды бота и сценарии

| Команда / действие | Что делает |
|--------------------|------------|
| `/start` | Запускает онбординг-мастер: выбор языка → выбор кампуса → передача телефона → приглашение отправить геолокацию. Повторный `/start` перезапускает мастер и обновляет профиль. |
| Выбор языка | Три кнопки: `English`, `Русский`, `O'zbek` (латиница). Дальнейший интерфейс бота — на выбранном языке. |
| Выбор кампуса | `Mirzo Ulugbek` / `Yashnobod` (кнопки с локализованными названиями). |
| Передача телефона | Кнопка «Поделиться контактом» (Telegram requestContact). Бот проверяет, что контакт принадлежит самому пользователю. |
| Отправка локации | Создаёт участника (waiting) в своём кампусе, запускает поиск **только среди участников того же кампуса**; при сборе группы — уведомляет всех троих на их языках. |
| `/leave` | Выход. Если был в группе — группа распадается, остальным приходит уведомление, они возвращаются в очередь и идёт перепоиск. В тест-режиме убирает последнюю локацию аккаунта. |
| `/status` | Показывает ваши локации и их статус (в очереди / в группе). |
| `/reset` | Удаляет все ваши локации. |

Валидация координат (диапазоны широты/долготы) выполняется до сохранения.
Если у пользователя нет профиля (онбординг не завершён), локация не принимается — бот
предлагает сначала пройти `/start`. Сбои отправки сообщений (например, пользователь
заблокировал бота) логируются и не роняют бота.

## Живая карта

`http://127.0.0.1:8080/live`

- Опрашивает `GET /data/state.json` каждые 5 секунд и перерисовывается.
- **Маркеры кампусов** (★, малиновый) — всегда отображаются на карте, даже без участников.
  Два кампуса: Мирзо Улугбек (41.356250, 69.373209) и Яшнобод (41.256928, 69.328708).
- **Группы** — цветом по группе, с пунктирными линиями к центроиду и номером
  группы; попап участника показывает имя, id, **телефон**, **кампус**, координаты.
- **Ожидающие** — серым; попап также показывает телефон и кампус.
- Сайдбар: «Групп: X · Ожидают: Y» + карточки групп с **кампусом** и **телефонами** участников
  (клик по карточке — зум к группе).
- Кнопка **🔄 Пересобрать группы** → `POST /rebuild`.
- `fitBounds` срабатывает только на первой загрузке — при автообновлении ваш
  текущий зум/позиция не сбрасываются.
- Имена и телефоны приходят из Telegram (недоверенные данные) и **экранируются** перед
  вставкой в HTML/попапы.

Безопасность HTTP-сервера: привязка только к `127.0.0.1`; отдаются лишь файлы из
`viewer/` и ровно два файла из `data/` (`output.json`, `state.json`) — `input.json`
и прочее закрыто (403); защита от path-traversal; CSP и `X-Content-Type-Options`.

- `store` — `joinAndMatch`, `leave` (распад + перегруппировка), `rebuild`,
  атомарное сохранение, фильтры по пользователю.
- `config` — валидация переменных окружения.
- `notify` — форматирование и устойчивость отправки.
- `viewer-server` — `POST /rebuild` → 200, `GET /rebuild` → 405, `input.json` → 403.

Браузерная часть (`live.js`) проверяется вручную, как и принято для вьюера.

## Наследие: батч-пайплайн

Проект вырос из разового скрипта кластеризации. Эти модули остались и не
используются ботом, но сохранены:

- `npm start` — читает `data/input.json`, группирует точки по 3, классифицирует
  по удалённости от фиксированной точки назначения (тиры 1/2/3), пишет
  `data/output.json`.
- `npm run viz` — отдельный батч-вьюер этого `output.json` (карта с тирами и
  звездой-назначением). Если запускаете вместе с ботом — задайте другой `PORT`,
  чтобы не было конфликта за 8080.

Живая карта бота (`/live`) и эта пересборка к ботовскому состоянию отношения к
батч-пайплайну не имеют.
